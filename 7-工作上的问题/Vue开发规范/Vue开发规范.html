# vue开发规范

# 基于模块开发

每一个组件（也就是模块）都是解决单一问题，独立的，可复用的，微小的，可测试的。如果一个组件做的太多事显得很臃肿，可以将其拆分为更小的组件。

# vue组件内部规范

# 组件名

组件名应该始终是多个单词，除了根组件App外

  • 语义化
  • 简短
  • 可读性
<!-- 推荐 -->
<app-header></app-header>
<user-list></user-list>
<range-slider></range-slider>

<!-- 避免 -->
<evaluation-btn></evaluation-btn> <!-- 虽然简短但是可读性差. 使用 `evaluation-button` 替代 -->
<button></button> <!-- button 太过于宽泛，在这里意义不明确 -->
<slider></slider> <!-- 与自定义元素规范不兼容 -->
# 组件数据

  1. prop

    Prop定义应该尽量详细，至少需要指定类型

    虽然Vue支持传递复杂的JavaScript对象通过props属性，但是应该尽可能的使用原始类型的数据（字符串、数字、布尔值），避免复杂的对象

    原因：

    • 使得组件API清晰直观
    • 使得其它开发者更好的理解每一个prop的含义

  2. v-for

    总是用key配合v-for，以便维护组件及其子树的状态。

  3. this赋值给component内的变量

    在组件中，this指向组件实例对象，为了在切换不同的上下文的时候，确保this都指向变量。

    原因：

    • 使用es6，就不再需要将this保存到一个变量中
    • 当使用箭头函数时，会保留this的作用域（this继承于上层作用域）
    • 如果你没有使用es6，当然也不会使用箭头函数，那必须把this保存到一个变量中。

  4. 组件结构化

    组件内钩子函数按照生命周期顺序写，易于阅读和理解

    <script>
	export defalut {
		// 组件的name 可应用于组件的跳转
		name: '',
    // 共享通用功能
		mixins: [],
    // 组成新的组件
    extend: {},
    // 使用其他组件
    components: {},
    // 组件属性、变量
    props: {}, // 按字母顺序
    // 生命周期钩子
    beforeCreated() {},
    mounted() {},
    // 路由钩子
    beforeRouterEnter() {},
    // 变量
    data() {},
    // 计算属性
    computed: {},
    // 方法
    methods: {},
    // 侦听器   
    watch: {}
	}
</script>
// 使用组件名作为样式作用域空间，组件内样式使用scoped属性，全局样式使用@import引入
<style scoped>
  @import 'xxx.scss'
</style>

  5. 组件事件命名

    Vue提供的处理函数和表达式都是绑定在ViewModel上的，组件的每一个事件都应该按照一个命名规范来

    • 事件名使用驼峰命名法，例如：fetchData
    • 事件命名应该以动词或者名词结尾
# 组件避免this.$parent

原因：

  • 组件必须相互保持独立，Vue组件也是
  • 如果一个组件需要访问其父组件的上下文，那么将不能在其他的上下文复用

解决：

  • 通过porps将传递给子组件
  • 通过props传递回调函数给子组件来达到调用父组件方法的目的
  • 通过在子组件触发事件来通知父组件
# 组件谨慎使用this.$refs

大部分情况下是可以避免refs来获取元素的上下文，当遇到props和events难以实现的功能时，通过this.$refs来实现，当需要操作DOM无法通过指令来做的时候可使用 this.$refs 而不是jquery、document.getElement*、document.queryElement。

# 指令都用缩写 （ ：表示 v-bind @ 表示 v-on）
# 计算属性的鲁棒性（Robust）考虑

在开发中，我们经常需要处理后端数据，修改、附加、删减，最后渲染到页面上，因为计算属性报错的话，页面会直接报错，直接影响用户体验，所以当我们对数据进行复杂的处理时，必须确保数据的存在和处理的拆分，可根据以下规则开发：

  • 数据嵌套不超过三层，数据的获取来源简化
  • 对数据尽量写简单的计算属性，如果有需求，也尽可能拆分
# 组件规范

  1. 组件分类

    模块统一放在modules文件夹下，统一放在components文件夹下，和父组件紧密耦合的子组件应该以父组件作为前缀命名，组件应该完整单词而不是缩写。

  2. 组件命名

    用 - 连接

  3. 图片命名

    • 图片后缀一律小写
    • 所有img元素必须加上alt属性，修饰性图片alt属性内容为空，内容性图片alt属性写相应的内容
    • 必须加上width，height属性，值为他的原大小，但不要用来对他进行缩放

  4. CSS

    • 必须以字母开头
    • 不允许单个字母的类选择器
    • 不允许带有广告等英文单词，防止该模块被浏览器当作垃圾广告过滤掉
    • 全小写并且使用 - 连接，不使用下划线 _
    • 禁止使用驼峰命名法

  5. JS

    • 禁止使用var
    • 变量使用let，采用驼峰命名法，首字母小写
    • 常量使用const，用下划线 _ 分隔开 全大写
    • 函数：首字母小写，驼峰命名法
    • 类以及命名空间：首字母大写，驼峰命名法
# 路由跳转规范

建议用name的方式 少用path方式跳转
